# Async Rollout

Async rollout is a pattern where generation, tool execution, reward computation, and trainer updates do not advance in lockstep. It is attractive for long-tail agentic workloads, but it turns policy versioning, cache isolation, delayed rewards, and observability into correctness contracts.

## Evidence Basis

- `source-areal-agentic-async-refs` records AReaL's source-reported fully asynchronous agentic RL positioning, rollout workflow fields including `versions`, staleness recovery, rollout pause/continue hooks, and version staleness metrics.
- `source-roll-ray-agentic-refs` records ROLL's source-reported sample-level/environment-level asynchronous rollout features, an async agentic config with `async_generation_ratio`, and role-separated actor/train/infer/reference config.
- `source-slime-agentic-buffer` records an independent asynchronous agent trajectory buffer pattern.
- `source-vllm-rollout-backend-refs` records vLLM async-RL cache and stale KV boundary documentation, useful when rollout backend state is part of the async contract.

## Use When

- The task objective names these frameworks or components.
- The acceptance criteria require explicit failure modes, rollback, and observability.
- The rollout and training surfaces have different latency, memory, or scaling needs.

## Design Checks

- Define the admission rule for stale samples: maximum `policy_version` lag, per-token `versions` interpretation, and rejection/weighting policy.
- Decide whether rollout can continue while weights update; if yes, define cache namespace, `flush_cache` or equivalent, and in-flight request behavior.
- Record rollout lifecycle events: generation start/end, reward start/end, trainer admission, weight update, pause/resume, cancellation, and replay.
- Bound long-tail tool/environment tasks with timeout, retry, cancellation, and partial-trajectory policy.
- Keep target-aware but not target-only context: AReaL and ROLL provide async/Ray patterns, while slime, verl, SGLang, and vLLM provide transferable data-buffer, mismatch, cache, and backend lessons.

## Failure Modes

- Trainer consumes samples generated by too-old policy versions.
- KV/prefix cache survives a weight update and contaminates replay or logprob comparisons.
- Reward delay hides a bad sample until after optimizer admission.
- Async queue drains in a different order from group construction, producing schema drift or missing group ids.
- Pause/resume hooks exist but are not covered by fault injection.

## Validation Ideas

- Version monotonicity test.
- Cache flush or cache namespace test.
- Rollout/train logprob mismatch smoke test.
- Failure injection for sync timeout or partial reward.
- Queue-drain test that mixes fast, slow, timeout, and rejected trajectories.
- Replay/recompute test across one accepted stale sample and one rejected stale sample.

## Open Gaps

- The cited evidence is source reading only; no local async rollout benchmark or distributed run was executed.
- Stale-policy thresholds are algorithm/task choices and must be set by the target task contract.
- Cross-framework adapter semantics still need implementation-specific code review.
